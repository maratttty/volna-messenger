# Проект: мессенджер Freeword

Имя продукта — Freeword (задаётся настройкой APP_NAME, не хардкодом). Цель: сначала рабочая веб-версия, потом нативные приложения iOS/Android. Рынок — Россия. Владелец не программист — объясняй простым языком; действия с оплатой, аккаунтами, телефоном, паспортом, Mac делает он по твоим инструкциям.

КЛЮЧЕВЫЕ ПРАВИЛА:
1. Веб-версия первая и в приоритете (работает без App Store/Google Play, без $99, без санкционных барьеров).
2. Реалистичные сроки: магазины — недели сверху (Google: тест 12 тестеров × 14 дней для нового личного аккаунта; Apple: одобрение + ревью + Mac). Не обещать нереального.
3. Юрисдикция: владелец в РФ (ИП). Российский аккаунт Apple не даёт монетизацию на заграницу, оплата с росс. карт не проходит. Глобальная монетизация — через иностранное юрлицо, отдельно и не в спешке. Веб от этого не зависит.
4. Шифрование честное, только реальная криптография. Никаких «нейросетей, шифрующих сообщения». E2EE — стратегический козырь (см. ниже), но пока НЕ реализован; до реализации НЕЛЬЗЯ заявлять «нас нельзя прочитать».
5. Приоритет — стабильность. Сначала «работает чётко без лагов», потом крупные новые фичи.

ВАЖНО — два разных понятия, не путать:
- E2EE (сквозное шифрование) = приватность, сервер не может прочитать переписку. Наш козырь, достижимо. Этап D.
- «Работает без VPN» / устойчивость к блокировкам = доступность (чтобы РКН не заблокировал). Это ОТДЕЛЬНАЯ и самая тяжёлая задача, постоянная гонка с государством, а не фича. Зарубежный хостинг от блокировок НЕ спасает (Telegram тоже за границей — его всё равно блокируют). На старте, пока продукт мал, VPN и так не нужен. Не обещать «без VPN» рано, браться поздно (этап F).

СТЕК (уже работает): React + Vite + TypeScript + Tailwind, Zustand, Supabase (Postgres + RLS + Realtime + Auth + Storage), деплой на публичный URL.

МЕТОД РАБОТЫ: задачи ПО ОДНОЙ, по порядку. Сначала кратко причина/план простым языком → жди моего «ок» → делай → собери проект (npx tsc -b && npx vite build) → отчитайся, как проверить → жди проверки перед следующей. НЕ ломай уже работающее. НЕ делай несколько крупных задач за раз. После каждой задачи обновляй PROGRESS.md, в конце сессии сохраняй в git.

ИНЖЕНЕРНЫЙ СТАНДАРТ: разделение слоёв (UI/состояние/логика/данные); права и валидация на сервере (RLS); целостность сообщений без потерь и дублей (оптимистичные обновления с откатом, идемпотентность по client_id, переподключение); ВСЁ мгновенно в реальном времени (отправка/удаление/редактирование/реакции/счётчики) без перезагрузки; секреты только в переменных окружения.

УТП/ПОЗИЦИОНИРОВАНИЕ: одна идея — «мессенджер, который физически не может прочитать вашу переписку, и это можно проверить». Честно про Telegram: у него E2EE только в секретных чатах 1-на-1, обычные чаты и группы видны его серверам; наше отличие (когда внедрим) — E2EE по умолчанию во всех чатах + открытый код. НЕ делать «детектор слежки». Не заявлять то, что ещё не реализовано.

ПЛАН (дорожная карта):
A. Стабилизация текущего веба (СЕЙЧАС): realtime/мгновенность, мобильный вид как настоящее приложение (PWA без «плавания»/зума), загрузка меню как в Telegram (кэш + «обновление…»), позиция прокрутки + кнопка «вниз», клавиатура на телефоне, кружки (индикатор только после тапа, голубой), меню сообщения строго над/под сообщением и не за краем, статусы в шапке чата, упоминания @ в группах, меню собеседника, стеклянная нижняя навигация, расширенные настройки.
B. Запуск веба: стабильность → свой домен → публичный URL → первые пользователи (друзья) → сбор багов. Установка через PWA.
C. Рост: пользователи → блогеры → аудитория РФ.
D. Флагман приватности: E2EE по умолчанию + открытый код → главный маркетинг.
E. Нативные приложения и магазины (после стабильного веба).
F. Устойчивость к блокировкам и монетизация — поздно и осторожно.
