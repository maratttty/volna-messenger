# Техническое задание — мессенджер Freeword

Документ для Claude Code. Это полный контекст проекта и план работ. Прочитай его целиком,
затем осмотри текущую кодовую базу и восстанови пошаговый план до конца.

---

## 1. Что строим и принципы

- Мессенджер для личного и группового общения. Ориентир по удобству — Telegram, но со своим
  именем, своими цветами и логотипом. Графику/иконки Telegram не копировать.
- **Сначала веб-версия**, потом нативные приложения iOS/Android.
- MVP = только ядро общения. Каналы, боты, звонки, платежи, премиум — поздние этапы.
- Качество важнее количества функций: лучше меньше, но стабильно и без потери сообщений.
- Владелец — не программист. Объясняй шаги простым языком. Перед действиями, требующими
  оплаты или регистрации внешних аккаунтов, предупреждай и говори, что именно сделать ему.

## 2. Модель безопасности (честные ожидания)

«Защищён как Telegram» = как обычные (не «секретные») чаты Telegram:
- Весь трафик только по HTTPS/WSS (TLS).
- Сообщения хранятся на сервере, база шифруется at rest.
- Строгая проверка прав на сервере: пользователь видит только свои чаты.
- Сессии с истечением и возможностью отзыва.
- Rate limiting на вход/отправку/инвайты, валидация ввода, проверка загрузок.
- Сквозного E2EE в MVP НЕТ (как обычные чаты Telegram). До реализации НЕЛЬЗЯ заявлять
  «нас нельзя прочитать». E2EE — стратегический козырь и главный маркетинг продукта,
  реализуется отдельным флагманским этапом D после стабилизации текущего продукта.
  Никакой «нейросети, шифрующей сообщения» — только реальная криптография.
- ВАЖНО: «E2EE / приватность» и «работает без VPN / обход блокировок» — РАЗНЫЕ вещи.
  Обход блокировок — отдельная тяжёлая задача (этап F), зарубежный хостинг от
  блокировок РКН не спасает. Не путать и не обещать рано.

## 3. Стек

- Фронтенд: React + Vite + TypeScript + Tailwind. С прицелом на дальнейшую сборку в мобильное
  приложение (React Native / Capacitor).
- Состояние: Zustand (чаты/сообщения) + Context (auth).
- Бэкенд/БД/realtime/auth/storage: Supabase (Postgres + RLS + Realtime + Auth + Storage).
- Деплой веба: GitHub Pages (https://maratttty.github.io/volna-messenger/), автодеплой через GitHub Actions при пуше в main.
- Все секреты — только в переменных окружения, не в коде/git.
- Имя продукта — настройка (APP_NAME), не хардкод.

## 4. Объём MVP (веб)

1. Регистрация/вход по email + пароль; обязательный уникальный username; сессии; выход;
   сброс пароля по email.
2. Профиль: display_name, username, аватар, статус «онлайн / был(а) недавно».
3. Поиск пользователя по username; контакты; старт личного чата.
4. Личные чаты 1-на-1 в реальном времени: статусы отправлено/доставлено/прочитано,
   «печатает…», время, разделители по датам, счётчик непрочитанных.
5. Группы до ~200: создание (название, аватар), добавление участников, инвайт-ссылка
   (с отзывом), роли владелец/админ/участник, системные сообщения, выход из группы.
6. Сообщения: текст, изображение, файл, голосовое, видео-кружок. Операции: ответить,
   редактировать своё, удалить (у себя/у всех), переслать.
7. Список чатов с последним сообщением и непрочитанными.
8. Поиск по чатам и по сообщениям внутри чата.
9. Веб-уведомления при закрытой вкладке; mute чата.
10. Тёмная тема по умолчанию; адаптив мобайл-фёрст; языки RU + EN.

Вне MVP: каналы, боты, звонки, E2EE-секретные чаты, стикер-маркет, оплаты, премиум, реклама.

## 5. Нижняя панель ввода и голосовые/кружки (по логике как Telegram, иконки свои)

- Поле ввода растёт вверх; слева скрепка (фото/файл); справа кнопка.
- Поле пустое → кнопка-микрофон; есть текст → кнопка «Отправить».
- Тап по кнопке записи переключает режим голосовое ↔ видео-кружок (иконка меняется,
  режим запоминается).
- Удержание кнопки → запись текущего режима; отпустил → отправка; свайп в сторону → отмена;
  во время записи — таймер и индикатор.
- Голосовое: дорожка с play, прогрессом и длительностью. Кружок: круглое видео с фронталки,
  играет по нажатию. Запись через getUserMedia/MediaRecorder, файлы в Supabase Storage.

## 6. Модель данных (ориентир, многое уже в supabase/schema.sql)

profiles, chats (direct/group), chat_members (роль, last_read_message_id, muted),
messages (type: text/image/file/voice/video_note/system, client_id для идемпотентности,
reply_to_id, edited_at, deleted), message_status (delivered/read), invites, attachments.
RLS на каждой таблице. client_id уникален по (chat_id, sender_id) — защита от дублей.

## 7. Инженерный стандарт

- Разделение слоёв: pages (рендер) / hooks (логика) / store (Zustand) / lib (доступ к данным) /
  components / types. Бизнес-логику не держать в UI-компонентах.
- Авторитет на сервере (RLS), клиент не доверяет себе.
- Целостность сообщений: без потерь и дублей; идемпотентная отправка; оффлайн-очередь;
  переподключение и догрузка пропущенного.
- Пагинация истории по курсору. Доставка <1 сек в норме.
- TypeScript, линтер, обработка ошибок и состояний загрузки/пустоты/ошибки.
- Тесты на критичную логику (отправка/доставка/права).

## 9. Критерии готовности MVP

Двое разных людей регистрируются и переписываются 1-на-1 (<1 сек, со статусами); группа 50+
без потерь/дублей; отправляются картинка/файл/голос/кружок; работают непрочитанные,
«прочитано», «печатает…»; приходят веб-уведомления; поиск находит чаты и сообщения;
корректное переподключение при обрыве сети; приложение не падает; веб доступен на публичном URL.

## 10. После MVP (магазины и направление развития)

- Приложения iOS/Android (общая кодовая база), пуши APNs/FCM. Сборка iOS требует Mac.
- Google Play: для нового личного аккаунта — закрытый тест 12 тестеров × 14 дней до продакшна
  (запускать рано) или аккаунт-организация. Apple: ~$99/год, для РФ — ограничение монетизации
  на зарубежную аудиторию и проблемы оплаты; вопрос юрисдикции решать отдельно с юристом.
  Веб от этого не зависит — запуск начинаем с веб-ссылки.
- Главное УТП (на будущее): E2EE по умолчанию во всех чатах + открытый код —
  «мессенджер, который физически не может прочитать вашу переписку, и это можно проверить».
  Честно про Telegram: у него E2EE только в секретных чатах 1-на-1, обычные чаты и группы
  видны его серверам. НЕ заявлять это до реализации E2EE.
